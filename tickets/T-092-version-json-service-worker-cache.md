# T-092 - Fix version.json service worker caching

## Goal

Make `/version.json` reliably return the live JSON on normal browser refreshes, not a stale service-worker or browser-cache response.

## Context

After T-090/T-091, production nginx serves `/version.json` correctly with `Cache-Control: no-store`, but Chrome can still show an old `Not Found` response on normal F5 while hard reload returns JSON. Production inspection showed `/sw.js` was cached by Cloudflare with `max-age=14400`, so clients could keep an old service worker that intercepts navigations before the fixed app shell is loaded.

## Contract

- The service worker script and registration script are served with `Cache-Control: no-store`.
- Workbox navigation fallback must not handle `/version.json`, `/api/*`, `/healthz`, or `/readyz`.
- `/version.json` remains generated by the web build and served by nginx with `no-store`.

## Acceptance criteria

- [x] `sw.js` and `registerSW.js` are served by nginx with `Cache-Control: no-store, no-cache, must-revalidate`.
- [x] Workbox navigation fallback denies `/version.json`, `/api/*`, `/healthz`, and `/readyz`.
- [x] The generated production service worker contains the denylist configuration.
- [x] The web build still emits `dist/version.json`.

## Implementation notes

The root cause is a stale service worker path, not missing production `version.json`. Hard reload bypasses enough client cache to hit nginx, while normal F5 can still be handled by the old SW/cache path.

- Inspect generated `apps/web/dist/sw.js` -> denylist excludes `/version.json`, `/api/*`, `/healthz`, `/readyz`
