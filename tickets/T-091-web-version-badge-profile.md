# T-091 - Move web version badge into profile

## Goal

Show the deployed web version only in the profile screen instead of globally on every route.

## Context

T-090 added automatic build metadata and exposed it in the app shell so deploy completion could be verified from the UI. After seeing it live, the badge is too visible for normal use and should move to a quieter location.

## Contract

The existing `VersionBadge` component stays unchanged and continues linking to `/version.json`. `AppShell` no longer renders it globally. `/profile` renders it at the bottom of the profile content.

### Endpoints / routes

| Method | Path       | Auth    | Success                                                 | Errors                  |
| ------ | ---------- | ------- | ------------------------------------------------------- | ----------------------- |
| GET    | `/profile` | session | Profile page includes the version badge near the bottom | existing profile errors |

### Request / response shapes

_n/a_

### Provides

A less prominent but still discoverable deployed-version indicator in the profile page.

### Consumes

- `VersionBadge` from `apps/web/src/components/layout/version-badge.tsx`
- `/version.json` generated by T-090

## Acceptance criteria

- [x] The version badge is not rendered by the global app shell sidebar or mobile floating layout.
- [x] The profile page renders the existing version badge at the bottom of the page.
- [x] The profile screen still typechecks, tests, and builds.

## Test matrix

| Case            | Input                           | Expected                                          |
| --------------- | ------------------------------- | ------------------------------------------------- |
| Profile view    | authenticated `/profile`        | version badge is visible at the bottom            |
| Other app views | authenticated non-profile route | no global version badge is rendered by `AppShell` |

## Files to touch

## Out of scope

## Implementation notes

Dokploy production debugging on 2026-07-11 found tag deployments were cancelled because Docker Swarm restarted Dokploy during parallel Compose builds on a small VM. The live Dokploy service now has `COMPOSE_PARALLEL_LIMIT=1` and `RELEASE_TAG=v0.29.11` set.
