# AGENTS.md

The contract every agent (and human) follows in CarNotea. Read it top to bottom
once; after that, jump to the **Task Router** for the specific thing you're
doing. Rules are written as commands, not prose — if a rule doesn't tell you
what to *do* or what to *run*, it's a bug in this file. Fix it.

CarNotea is a vehicle-diary PWA. Stack and rationale: [`README.md`](./README.md)
and [`docs/adr/`](./docs/adr/). This file is about *how we work*, not *what we
build*.

## Always

- Find a ticket before non-trivial work. No ticket → create one from
  [`tickets/_template.md`](./tickets/_template.md) first. ("Non-trivial" = more
  than a typo.)
- Read the closest `AGENTS.md` to the file you're editing — area files
  (`apps/*/AGENTS.md`, `packages/*/AGENTS.md`) override this root for their
  subtree.
- Match the surrounding code's style, naming, and structure over your own
  defaults. See [`docs/conventions.md`](./docs/conventions.md).
- Validate every external input (HTTP body/query/params, env, third-party JSON)
  with **Zod** at the boundary. Derive types with `z.infer` — never write a
  parallel TS type by hand.
- Keep every user-facing string in **both** `pl` and `en` translation files.
  See [ADR-0007](./docs/adr/0007-i18n-pl-en.md).
- Run the smallest relevant set of **Validation Commands** (below) before you
  call a task done.
- After a human corrects you, append a rule to
  [`docs/agents/lessons.md`](./docs/agents/lessons.md) so the mistake doesn't
  recur. One correction → one lesson.
- Update docs in the *same* change: a new command → `docs/getting-started.md`;
  a new package → `docs/tech-stack.md`; a new top-level folder → the repo map
  here and in `README.md`.

## Ask First

- Before reducing a ticket's scope, changing its architecture, or splitting it.
- Before adding a top-level dependency, a build tool, or a runtime — that needs
  an [ADR](./docs/adr/) (or an update to one).
- Before changing a public contract: an HTTP route shape, a DB column/trigger,
  a shared Zod schema in `@carnotea/shared`, an env-var name.
- Before touching more than one app/package in a way no ticket describes.
- Before running destructive DB commands (`pnpm db:reset`, dropping a schema)
  against anything but your own throwaway local database.

## Never

- Never mark a ticket `done` when its acceptance criteria aren't all true. Half
  done stays `in_progress`. See [§ Be honest](#be-honest).
- Never edit a generated file by hand (anything `*.gen.ts`, `*.d.ts`,
  lockfiles). The Drizzle migration files under `packages/db/migrations/` are
  generated by `pnpm db:generate` — do not edit them manually; change the
  schema in `packages/db/src/schema/` instead. See
  [ADR-0002](./docs/adr/0002-drizzle-schema-as-code.md).
- Never use `class-validator` or a second validation layer. Zod only.
- Never hard-code a user-facing string in JSX/TSX. Route it through i18n.
- Never add speculative code: no abstractions "for later", no fallbacks for
  impossible states, no feature flags before a second behaviour exists, no
  error handling for errors that can't happen.
- Never edit an accepted ADR or a merged ticket file — supersede it with a new
  one instead.
- Never commit secrets, a real `.env`, tokens, or keys.

## Validation Commands

Run the smallest set that covers your change. From the repo root:

```bash
pnpm lint           # ESLint across affected packages (Turborepo-cached)
pnpm format:check   # Prettier check
pnpm typecheck      # tsc across affected packages
pnpm test           # Vitest unit/integration
pnpm build          # build affected packages
pnpm lint:ws        # sherif — workspace dependency consistency
```

> Bootstrap note: until the first app/package tickets land, these are wired but
> mostly no-ops (nothing to lint/build yet). That's expected, not a failure.

### UI verification with agent-browser

When a ticket touches `apps/web` or any UI component, also verify it in a real
browser. `agent-browser` is installed globally and configured via `agent-browser.json`
at the repo root. Chrome for Testing is bundled — no system browser needed.

```bash
pnpm --filter @carnotea/web dev          # start the dev server first
agent-browser open http://localhost:5173 # navigate to the app
agent-browser snapshot -i                # list interactive elements + refs
agent-browser chat                       # natural-language control (interactive)
```

See `docs/agents/self-review.md` §UI verification for the full checklist.

A change is **done** when: every acceptance-criterion box in the ticket is
genuinely checked, the relevant commands above pass, UI changes are verified in
agent-browser, and the docs touched by the change are updated in the same commit.

## Skills / commands

The same three actions are wired up for every environment, all backed by the
one environment-agnostic spec in
[`docs/agents/ticket-execution.md`](./docs/agents/ticket-execution.md):

| Action | What it does | Claude Code | Codex | Other |
| ------ | ------------ | ----------- | ----- | ----- |
| Pick next | Recommends the next unblocked ticket. | `/next-ticket` | `next-ticket` skill | read the doc |
| Work a ticket | Analyze → plan (human approval) → implement in a worktree → self-review → report. | `/work-ticket <id>` | `work-ticket` skill | read the doc |
| Smart commit | Groups uncommitted changes into logical commits with Conventional Commit messages, waits for approval, then commits. Does not push. | `/smart-commit` | `smart-commit` skill | `.claude/commands/smart-commit.md` |
| Ship a PR | Validates, pushes the branch, opens a PR. | `/ship-pr` | `ship-pr` skill | read the doc |

- **Claude Code** loads these as slash commands from `.claude/commands/*.md`.
- **Codex** loads them as project skills from `.codex/skills/<name>/SKILL.md`
  (scanned at session start). Codex skills are checked into the repo; Codex's
  older `~/.codex/prompts/` custom prompts are global-only and deprecated, so we
  don't use them.
- **Any other agent or a human** follows
  [`docs/agents/ticket-execution.md`](./docs/agents/ticket-execution.md)
  directly. The skill/command files are thin pointers to it — keep behaviour in
  the doc, not duplicated across environments.

> New command files only appear after restarting the agent (Claude Code or
> Codex) or opening a new chat — they're scanned at startup.

## Hierarchy

For a file at path `X`, AGENTS.md files apply in order, later overriding earlier:

1. **Root** (this file) — everywhere.
2. **Area** `apps/<app>/AGENTS.md` or `packages/<pkg>/AGENTS.md`.
3. **Sub-area** `apps/<app>/src/<module>/AGENTS.md` if present.

No scoped file → the closest parent applies. Found a recurring gotcha in a
sub-area? Create an `AGENTS.md` there rather than bloating this root.

`CLAUDE.md` is a one-line redirect to this file (`@AGENTS.md`) so Claude Code
loads it automatically. Don't put rules in `CLAUDE.md` — they'd drift from here.

## Principles

- **Simplicity first.** The simplest change that fully solves the ticket wins.
- **Root cause, not patch.** No temporary hacks. Fix the actual problem.
- **Minimal blast radius.** Touch only what the ticket needs.
- **One ticket, one PR.** Discovered follow-up work → a new ticket, not scope
  creep.

## Be honest

- A skipped test is reported as skipped.
- A UI change that wasn't visually verified is reported as not verified.
- A ticket that's half-done stays `in_progress`. Never `done`.
- If a command failed, say so and show the output. Don't claim green when it's
  red.

```
│   ├── typescript/       # shared tsconfig presets
│   └── github/           # reusable GitHub Actions setup

│   ├── architecture.md
│   ├── conventions.md
│   ├── getting-started.md
│   └── tech-stack.md
├── tickets/              # markdown tickets that drive the work
│   ├── _template.md  INDEX.md  README.md  T-<id>-<slug>.md

│   └── commands/         # Claude Code slash commands (/next-ticket, /work-ticket, /ship-pr)

│   └── skills/           # Codex skills — same three actions for Codex CLI

```

`apps/*`, `packages/*`, and `tooling/*` are created by their scaffold tickets —
they don't all exist yet. That's normal; the tickets build them in order.
